Hoe bouwen we een veilige software? 

Hierbij kunnen de meeste voorkomende kwetsbaarheden gebruiken van het OWASP.org.

1. Injections:
  Hier sturen we code of valse informatie in een database.
  Bijvoorbeeld:
    In een input field of dataset.

  Hoe kunnen we dit tegen werken?
    Gebruik zowel frontend als backend validatie op input.
    Hiervoor is regular expressie een goede tool.
    Kijk ook steeds of de data dat je krijgt klopt.

2.Niet werkende authenticaties:
  Login mogelijkheden die kunnen omzeilt worden of niet veilig zijn is onveilig.
  Bijvoorbeeld:
    Als een hacker snooped op het netwerk waar een gebruiker zich inlogt op een pagina die niet via SSL is ge-encrypteerd.
    Dit wilt zeggen dat de inhoud van die pagina ook beschikbaar is voor wie meeluistert op het netwerk.

  Hoe kunnen we dit tegen werken?
    Gebruik altijd HTTPS. Gebruik multi-factor authentications. Session timeouts, secure cookies, ...

3.Sensitive Data Exposure:
  Data dat veilig moet staan, moet dan ook op een veilige manier op een veilge plaats terecht komen. Dit is zeker het geval bij paswoorden.
  Zorg dus ook dat de database ge-encrypteerd staat op een veilige plaats. Als de database publiekelijk toegankelijk is, moet dit zeker een login en passwoord op staan.

  Bijvoorbeeld:
    Paswoorden die worden opgeslagen op een database moeten nooit in plain text staan in een database.
    Dit moet ge-hashed worden. Dit is een wiskundige algoritme dat een text naar een hashstring verandered.
    De volgende keer als een gebruiken zich inlogd, word dat passwoord gehashed (met hetzelfde aantal salt) en vergeleken met dat in de database.

  Hoe kunnen we dit tegen werken?
    Zoals bij het voorbeeld, passwoorden nooit in plain text opslagen. Databases op een veilige plaats zetten, encrypteren en beveiligen met een login.

4.XML external entities:
  Dit zijn aanvallen/code injecties via XML parsers.
  XML parsers zorgen ervoor dat een stuk string of dergelijke wordt omgevormd
  	naar XML bestand. Een XML bestand is een Markup bestand met data (daarvan
  de X) .

  Bijvoorbeeld:
    In een XML bestand kunnen kleine stukken code staat, zoals instellingen.
    Een hacker kan door een verouderde parser, deze code verouderen door 	commando’s.

  Hoe kunnen we dit tegen werken?
    JSON gebruiken oftwel een betere en/of nieuwere xml parser zoeken.

5.Broken access control
  Pagina’s dat achter een paywall of loginwall zitten moeten ook beschermd worden.
  Als deze niet beschermd worden kunnen we daar gewoon naar surfen zonder ernaar verwijst te worden via een login.

  Bijvoorbeeld:
    Een admin pagina kan /admin zijn. Nergens staat er ergens een link naar.
    Maar als je het weet of probeerd naar te surfen, en er staan geen login achter. Kan dit voor problemen zorgen.

  Hoe kunnen we dit tegen werken ?
    Controleer wie wat beheerd, inloggen en users bijhouden, gebruik sessies voor het gebruik van de user te onthouden waneer hij heeft ingelogd

6.Security misconfiguration:
  Als servers niet goed beveilig is en zich blootstelt aan alle gebruikers kan het 	heel onveilig zijn.
  Dit zijn vooral errors, bestanden, en serverinfo.

  Bijvoorbeeld:
    Een error 500 bestaat er voor een reden, maar als die niet word opgevangen 	word de exacte error van de server geschreven naar de html pagina.
    Daar staat in welke code wordt gebruikt en welke bestanden

  Hoe kunnen we dit tegen werken?
    Error code opvangen, geen bestanden die niet nodig zijn publiek zetten, ...

7.Cross Site Scripting XSS
  Hierbij sturen we code waar we code niet verwachten. Waardoor men scripts kan zetten in een database.
  Bijvoorbeeld:
    In een input field.

  Hoe kunnen we dit tegen werken?
    Gebruik zowel frontend als backend validatie op input.
    Hiervoor is regular expressie een goede tool.

8.Insecure desalization:
  Deserialization kan een aanvaller in staat stellen om code in de applicatie van op afstand uit te voeren,
  geserialiseerde objecten te wijzigen/verwijderen, of injectie aanvallen uit te voeren.
  Hoe kan je dit voorkomen? Implementeren van integriteitscontroles zoals digitale handtekeningen op geserialiseerde objecten.

9.Using components with known vulnerabilites:
  Als bepaalde functies van een bepaald systeem niet meer veilig zijn, best niet 	gebruiken dan.

  Bijvoorbeeld:
    Een functie van javascript dat duidelijk niet meer mag gebruikt worden.
    Deze kan worden g gebruikt op veilige data op te halen, of commando’s door te 	sturen.

  Hoe kunnen we dit tegen werken?
    Altijd kijken dat third-party functies veilig zijn en mogen gebruikt worden.
    En kijk op de oficiele bronnen welke functies niet meer mogen gebruikt worden.

10.Veel webapplicaties ondernemen onvoldoende stappen om datalekken op te sporen.
 Dit zet aanvallers in staat om regelmatig over te stappen naar andere systemen en bedreigingen.
 Je kan niet aannemen dat de preventiemaatregelen hun werk doen, dus raad OWASP aan om voldoende logging/monitoring te doen,
 en een plan te hebben voor wanneer er iets misgaat.

Beschrijving SLDC toepassing in project

SDLC van ons project:

Hier gaan we over de System Development Life Cycle voor ons geïntegreerd project.We hebben elk van de 6 fases uitgelegd wat en hoe we ze gaan uitwerken.

Planning-analyse:

In de aula werden er groepen gekozen, we zijn dan samengekomen tot: Thomas, Tim, Jaimie, Joni en Karel. Nadien hebben we als groep besloten welke communicatiemiddelen we gaan gebruiken voor aan ons project te werken (discord, messenger, ...). We hebben besproken op welke momenten we hieraan zouden werken. Nadien is beslist om samen te komen op de campus zodat we ons zo goed mogelijk konden focussen op het project.Bij onze eerste samenkomst hebben we dan ook geanalyseerd hoe we gaan werken, en gebrainstormd over de toekomstige taken die ons te wachten staan binnen het project. Er is ook een taakverdeling opgesteld om het werk evenredig te verdelen. Dit ook zodat alle leden binnen het project te allen tijde weten waar ze aan kunnen werken.

Design:

Na de planningsanalyse zijn we op de eerste samenkomst aan het ontwerp begonnen. We hebben samen nagedacht over wat we allemaal gaan toepassen, en ervoor gezorgd dat iedereen dit begreep. We besloten om de database te hosten, omdat dit voor ons de meest praktische oplossing bleek te zijn.We vonden dit een belangrijke stap vooraleer we aan het project begonnen zijn. Dit zodat we wisten dat iedereen op hetzelfde niveau zit, en dat we allemaal evenveel bijdragen leveren aan het project.

Develop

Na de designfase hebben we hetgeen toegepast dat in de projectlessen is aangehaald. Dit hebben we ook in groep gedaan om zo een goede samenwerking op punt te stellen.Vooraleer we begonnen met het schrijven van de code, hebben we testen geschreven voor ons programma. Dit omdat het ons achteraf veel werk zou besparen. Dit hebben we eveneens in groep gedaan, om een snelle voorruitgang te boeken. Zo wist iedereen op wat er getest ging worden, wat het nut is van deze testen en ook waar we in te toekomst rekening mee moeten houden.Nadien hebben we ervoor gezorgd dat iedereen hun eigen taak heeft bij het project. Dit werd telkens nagekeken door een ander teamlid.

Implementation

Nadat de meeste code is geschreven in de develop fase, gaan we het project in productie zetten.Ons project zal dan ook gehost worden op een onlineserver. Hiervoor moeten we zorgen dat er voldoende security maatregelen en controles zijn. Zodat zowel de server van de site en de database veilig online staan.

Maintain:

Aan maintain denken we momenteel dat we niet veel gaan doen. We verwachten dat dit project alleen maar dit semester is en we dus echter niets voor maintain moeten of kunnen doen.Echter, als dit niet zo is kunnen we altijd een persoon aanduiden voor de applicatie te onderhouden.

Mijn digitale fingerprint

Dit is mijn digitale fingerprint. Aangezien ik bijna enkel Linux (Manjaro) gebruik wordtt ik al veel minder ge-tracet dan een normale gebruiker. Maar ander maatregelingen naam ik ook:

  • Ik gebruik constant een VPN (virtual private network)
  • Ik gebruik firefox, met de strengste settings. ik blokeer alle third-party cookies
  • ik gebruik de volgende addons:
    • Noscript: blokeert alle javascripts en XSS attacks.
    • HTTPS everywhere: Zorgt ervoor dat ik enkel https sites bekijk
    • Privacy badger: Blokkeer alle trackers online
    • decentraleyes: ipv CDN's online te gebruiken, gaat deze addon het lokaal zetten. voor betere privaivacy
  • Mijn cache en cookies worden ook telkens verwijdert nadat ik firefox afsluit. Enkel bepaalde sites die ik gekozen heb zullen niet verijwdert worden
  • Online accounts van Microsoft en google heb ik verwijder. Door mijn privacy te verhogen
  • Zelfs als besturingsysteem van mijn smartphone gebruik ik LineageOS, dit is een custom Android OS Speciaal voor privacy gemaakt
  • Ook gebruik ik een passwoord manager, voor gevallen als data breaches.
